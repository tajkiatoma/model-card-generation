## Model Details
This section provides fundamental information about the model, helping stakeholders understand its context and key characteristics.

### Person or organization developing model:
The model was developed by Dongfu Jiang, Xiang Ren, and Bill Yuchen Lin. Their affiliations include the Allen Institute for Artificial Intelligence, the University of Southern California, and Zhejiang University (2306.02561.pdf, p. 1).

### Model date:
The academic paper describing the model is dated June 30, 2023 (2306.02561.pdf, p. 1).

### Model version:
The model is described in version 3 of the academic paper "LLM-BLENDER: Ensembling Large Language Models with Pairwise Ranking and Generative Fusion" (2306.02561.pdf, p. 1). This model, named PAIRRANKER, is a specialized pairwise comparison method designed to distinguish subtle differences between candidate outputs from various Large Language Models (LLMs) (2306.02561.pdf, p. 1). It differs from pointwise ranking methods (like the reward model in InstructGPT) by jointly encoding a pair of candidates for direct comparison, which is argued to be more effective for discerning subtle differences between high-quality LLM outputs (2306.02561.pdf, p. 4).

### Model type:
PAIRRANKER is a pairwise ranking model (`"ranker_type": "pairranker"`) designed to score and rank text generations (ranker_config.json.txt).

*   **Architecture**: The model is a cross-attention Transformer that jointly encodes an input text and a pair of candidate outputs to determine the superior one (2306.02561.pdf, p. 1, 4). The backbone model is `microsoft/deberta-v3-large` (ranker_config.json.txt, key: "model_name"; 2306.02561.pdf, p. 14), which has 400 million parameters (2306.02561.pdf, p. 7). The architecture also employs a five-layer multi-layer perceptron (MLP) with a hyperbolic tangent activation function to process embeddings and compute scores (2306.02561.pdf, p. 14).
*   **Size**: The backbone DeBERTa model has 400M parameters (2306.02561.pdf, p. 7).
*   **Context Length**: The model supports a maximum length of 1224 tokens for the source input and 412 tokens for the candidate input (ranker_config.json.txt, keys: "source_maxlength", "candidate_maxlength").

### Training details:
The model was trained using a supervised, pairwise comparison approach (2306.02561.pdf, p. 4).

*   **Algorithm**: The model is trained to compare pairs of candidate responses for a given input and predict which one is better. During training, a sub-sampling strategy is used where 5 pairs of candidates are randomly selected per input to ensure learning efficiency (2306.02561.pdf, p. 5). The order of candidates within each training pair is shuffled to help the model learn to be consistent (2306.02561.pdf, p. 6).
*   **Loss Function**: The learning problem is treated as a multi-task classification problem (2306.02561.pdf, p. 5). The appendix notes that a simple Binary Cross-Entropy (BCE) loss was found to be effective (2306.02561.pdf, p. 14). The configuration file specifies the loss type as "instructgpt" (ranker_config.json.txt, key: "loss_type").
*   **Optimizer**: The Adafactor optimizer was used (2306.02561.pdf, p. 14).
*   **Hyperparameters**:
    *   Epochs: 5 (2306.02561.pdf, p. 14).
    *   Batch Size: 64 (2306.02561.pdf, p. 14).
    *   Learning Rate: A maximum learning rate of 1e-5 with a 5% warm-up ratio and a linear learning rate scheduler (2306.02561.pdf, p. 14).
    *   Dropout: 0.05 (ranker_config.json.txt, key: "drop_out").
    *   Precision: Training was done with FP16 precision (ranker_config.json.txt, key: "fp16").

## Intended Use
This section outlines the intended applications of the model.

### Primary intended uses:
The primary use of PAIRRANKER is to rank a set of candidate outputs generated by multiple Large Language Models (LLMs) for a given input instruction (2306.02561.pdf, p. 1). It is a core component of the LLM-BLENDER framework, which is an ensembling method designed to leverage the diverse strengths of different LLMs to produce consistently superior responses (2306.02561.pdf, p. 2).

The model is specifically designed to effectively discern subtle differences between high-quality candidate outputs, a scenario where traditional pointwise scoring methods may be less effective (2306.02561.pdf, p. 2, 4). By selecting the best-ranked candidate for each input, the model facilitates a dynamic ensembling approach that outperforms any single individual LLM (2306.02561.pdf, p. 2).

The input to the model is a pair of candidate texts (`candidate1`, `candidate2`) along with the original source text (`source`). The model outputs scores indicating which candidate is superior (2306.02561.pdf, p. 5).

### Primary intended users:
The primary intended users are researchers and practitioners in the field of Natural Language Processing who are deploying, studying, or developing LLMs, particularly those interested in ensemble learning techniques to improve model robustness, generalization, and accuracy (2306.02561.pdf, p. 2).

### Out-of-scope uses:
The model is a ranker, not a text generator. It is not intended to create new text. Its function is to evaluate and rank a pre-existing set of candidate texts (2306.02561.pdf, p. 2). The performance of PAIRRANKER is inherently limited by the quality of the candidate pool it is given; it cannot produce a high-quality output if none is provided in its input candidates (2306.02561.pdf, p. 6).

## How to Use
This section outlines how to use the model.

The model operates by comparing pairs of candidates for a given source input.

**Input Structure:**
The model takes a single concatenated sequence as input, formed with special separator tokens. The format is:
`<s><source> x </s> <candidate1> yi </s> <candidate2> yj </s>`
where `x` is the source input text, and `yi` and `yj` are the two candidate texts being compared (2306.02561.pdf, p. 5).

**Inference Process:**
During inference, the model is run on all pairs of candidates to produce a comparison matrix `M`, where `Mij` represents the model's confidence that candidate `yi` is better than `yj` (2306.02561.pdf, p. 6). Based on this matrix, a final ranking of all candidates is determined using an aggregation method. Three methods are proposed:

1.  **MaxLogits**: This method calculates a score for each candidate by summing the logit differences against all other candidates. It is the best-performing but most computationally intensive method, requiring O(N²) comparisons for N candidates (2306.02561.pdf, p. 6).
2.  **MaxWins**: This method calculates a score for each candidate based on the number of "wins" in its pairwise comparisons against other candidates (2306.02561.pdf, p. 6).
3.  **Bubble Sort (`bubble`)**: This is an efficient aggregation method that performs a single bubble sort run, requiring only N-1 comparisons to identify the best candidate. This is the default inference mode specified in the configuration (`"inference_mode": "bubble"`) and offers a strong trade-off between performance and efficiency (ranker_config.json.txt, key: "inference_mode"; 2306.02561.pdf, p. 6).

## Factors
This section addresses variables that may impact the model's performance.

### Relevant factors:
The primary relevant factor is the diversity of the LLMs used to generate the candidate outputs. The model's development is motivated by the observation that different LLMs have varying strengths and weaknesses due to differences in their training data, architectures, and hyperparameters. The optimal LLM for a given input can vary significantly, and this model is designed to navigate that variance (2306.02561.pdf, p. 1). The nature of the input instruction or task is also a key factor, as the model was evaluated on a mixture of instruction-following tasks (2306.02561.pdf, p. 3).

### Evaluation factors:
The model is evaluated on its ability to rank candidates from a pool of 11 different open-source LLMs (e.g., Vicuna, OpenAssistant, Alpaca) (2306.02561.pdf, p. 3). The evaluation is performed on the `MixInstruct` benchmark, which contains a variety of instruction-following tasks, thereby testing the model's performance across different types of instructions (2306.02561.pdf, p. 3).

## Metrics
This section describes how the model's performance is evaluated.

### Model performance measures:
The model's performance is assessed using two main categories of metrics:

1.  **Conventional NLG Metrics**: These are used to measure the quality of the top-ranked candidate selected by the model. The specific metrics used are BERTScore, BARTScore, and BLEURT (2306.02561.pdf, p. 6).
2.  **GPT-Rank**: This is a ranking-based metric created for the evaluation. It involves using ChatGPT to perform pairwise comparisons of all candidate outputs for a given input. The final rank of a candidate is determined by its number of "wins" in these comparisons (i.e., MaxWins aggregation). A lower average rank indicates better performance (2306.02561.pdf, p. 6-7).
3.  **Ranking Correlation Metrics**: To evaluate the quality of the entire ranking produced by PAIRRANKER (not just the top choice), its rankings are compared against the oracle ranking from GPT-Rank. The metrics used for this comparison are Pearson Correlation, Spearman's Correlation, and Spearman's Footrule distance (2306.02561.pdf, p. 8).

### Decision thresholds:
The model does not use explicit decision thresholds. Instead, it makes decisions by comparing the output scores for a pair of candidates. For instance, in the MaxLogits aggregation method, the relative scores determine the ranking (2306.02561.pdf, p. 6).

### Variation approaches:
The model's performance is compared against several baselines, including random selection and other ranking methods like MLM-Scoring, SimCLS, and SummaReranker. The evaluation is conducted on the `MixInstruct` test set, and results are aggregated across all examples to produce the final metrics (2306.02561.pdf, p. 7, Table 2).

## Evaluation Data
This section provides details about the datasets used to evaluate the model.

### Datasets:
The model was evaluated using the test split of the **MixInstruct** benchmark dataset, which was introduced in the same paper (2306.02561.pdf, p. 2). The test set consists of 5,000 examples (2306.02561.pdf, p. 3, Table 1). For each example in the test set, candidate outputs were generated from 11 popular open-source LLMs. Oracle pairwise comparisons for these candidates were obtained by prompting ChatGPT to serve as a judge (2306.02561.pdf, p. 2-3).

### Motivation:
The MixInstruct dataset was created to provide a large-scale benchmark for evaluating LLM ensembling methods on instruction-following tasks. It addresses the need for a standardized way to assess methods that rank and fuse outputs from multiple LLMs (2306.02561.pdf, p. 2).

### Preprocessing:
For each of the 5,000 test instructions, N=11 candidate outputs were generated using various open-source LLMs (e.g., Vicuna, OpenAssistant, Alpaca, MPT) (2306.02561.pdf, p. 3). These candidates form the pool that the PAIRRANKER model evaluates.

## Training Data
This section provides details about the datasets used to train the model.

### Datasets:
The model was trained on the training and validation splits of the **MixInstruct** dataset (2306.02561.pdf, p. 3). This comprises 100,000 training examples and 5,000 validation examples (2306.02561.pdf, p. 3, Table 1). The dataset is a curated collection of instruction examples from four main open-source sources (2306.02561.pdf, p. 3):
*   **Alpaca-GPT4**: 22,862 examples
*   **Dolly-15K**: 7,584 examples
*   **GPT4All-LAION**: 76,552 examples
*   **ShareGPT**: 3,002 examples

### Motivation:
The dataset was chosen to train the ranker on a large and diverse set of instruction-following examples, enabling it to learn how to compare outputs across a wide range of tasks (2306.02561.pdf, p. 3).

### Preprocessing:
1.  **Candidate Generation**: For each of the 110,000 instructions in the training and validation sets, N=11 candidate outputs were generated using the same set of popular open-source LLMs as for the evaluation data (2306.02561.pdf, p. 3).
2.  **Label Generation**: Unlike the test set, ground-truth labels for the training/validation sets were not generated by ChatGPT. Instead, the quality of each candidate `yi` was estimated by its similarity to the ground-truth response `y` from the original dataset. This similarity was measured using conventional metrics like BERTScore, BLEURT, and BARTScore (2306.02561.pdf, p. 4). BARTScore was ultimately chosen to provide supervision for training due to its high correlation with GPT-Rank (2306.02561.pdf, p. 7, 8).

## Quantitative Analyses
This section presents disaggregated evaluation results.

### Unitary results:
The performance of PAIRRANKER was evaluated on the MixInstruct dataset and compared against individual LLMs and other ranking baselines (2306.02561.pdf, p. 7, Table 2).

*   **GPT-Rank**: PAIRRANKER achieved an average GPT-Rank of 3.20. This significantly outperforms the best individual LLM, OpenAssistant (3.90), and other rankers like SimCLS (3.50) and SummaReranker (3.66).
*   **NLG Metrics**: The selections made by PAIRRANKER also scored highly on conventional metrics, achieving a BARTScore of -3.14, which was better than any individual LLM.
*   **Top-3 Performance**: PAIRRANKER's selected output ranked in the top 3 for 65.12% of examples, compared to 52.88% for the best individual LLM (Vicuna).
*   **Ranking Correlation**: PAIRRANKER's rankings showed the highest correlation with the oracle GPT-Rank across all metrics: Pearson Correlation (46.98), Spearman's Correlation (44.98), and Spearman's Footrule distance (27.52) (2306.02561.pdf, p. 8, Table 3).

## Caveats and Recommendations
This section lists unresolved issues and provides guidance for users.

### Caveats:
*   **Computational Cost**: To achieve the best performance (using the MaxLogits method), the model must be called O(N²) times for N candidates, which can be computationally expensive. While these inferences can be run in parallel, the total compute is high (2306.02561.pdf, p. 9, 10).
*   **Evaluation Limitations**: The evaluation relies on automatic metrics and ChatGPT as a proxy for human judgment. The authors acknowledge that large-scale human evaluation was not feasible but would provide more reliable results (2306.02561.pdf, p. 10).
*   **Dependency on Candidate Pool**: The effectiveness of PAIRRANKER is fundamentally constrained by the quality of the candidate outputs it receives. It can only select the best option from the provided pool and cannot generate a better response if one is not present (2306.02561.pdf, p. 6).

### Recommendations:
*   **Efficiency vs. Performance Trade-off**: Users should choose an inference method based on their needs.
    *   For high efficiency, the **`bubble`** inference mode is recommended. It requires only N-1 comparisons and achieves strong performance (2306.02561.pdf, p. 6, 17).
    *   For maximum performance, the **`MaxLogits`** method is recommended. The computational cost can be managed by executing the pairwise comparisons in parallel (2306.02561.pdf, p. 6, 10).
*   **Future Research**: The paper suggests that future work could explore extending the framework to other modalities (beyond text), developing more sophisticated ranking and fusion techniques, and investigating the transferability of the approach to other domains (2306.02561.pdf, p. 9).

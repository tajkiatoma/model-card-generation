## Model Details
This section provides fundamental information about the model, helping stakeholders understand its context and key characteristics.

### Person or organization developing model:
The model was developed by Dongfu Jiang, Xiang Ren, and Bill Yuchen Lin from the Allen Institute for Artificial Intelligence, University of Southern California, and Zhejiang University (LLM-BLENDER_paper.pdf, p. 1).

### Model date:
The associated academic paper is dated June 30, 2023 (LLM-BLENDER_paper.pdf, p. 1).

### Model version:
This model is the PAIRRANKER module of the LLM-BLENDER framework. LLM-BLENDER is an ensembling framework designed to combine the outputs of multiple Large Language Models (LLMs) to achieve superior performance. It consists of two modules: PAIRRANKER for ranking candidate outputs and GENFUSER for fusing the top-ranked candidates into a final output (LLM-BLENDER_paper.pdf, p. 1). This model card specifically describes the PAIRRANKER module, which is trained to perform pairwise comparisons of text generated by different LLMs (LLM-BLENDER_paper.pdf, p. 4).

### Model type:
The model is a pairwise ranking model, referred to as PAIRRANKER (config.json; LLM-BLENDER_paper.pdf, p. 2).
*   **Architecture:** It is built upon a `microsoft/deberta-v3-large` backbone, which is a Transformer-based cross-attention encoder (config.json; LLM-BLENDER_paper.pdf, p. 2, 14). The model has approximately 400 million parameters (LLM-BLENDER_paper.pdf, p. 7). It employs a five-layer multi-layer perceptron (MLP) with a hyperbolic tangent activation function on top of the Transformer embeddings (LLM-BLENDER_paper.pdf, p. 14).
*   **Functionality:** It is a text classification model that takes a source text and a pair of candidate texts as input and determines which candidate is superior (LLM-BLENDER_paper.pdf, p. 2).
*   **Tokenizer:** It uses a `DebertaV2Tokenizer` with a vocabulary size of 128,000 (tokenizer_config.json; tokenizer.json).
*   **Context Length:** The model supports a maximum source length of 1224 tokens and a maximum candidate length of 412 tokens (config.json).
*   **Special Tokens:** The tokenizer includes standard special tokens like `[CLS]`, `[SEP]`, `[PAD]`, `[UNK]`, and `[MASK]` (special_tokens_map.json), as well as added special tokens for structuring the input: `<|source|>`, `<|candidate1|>`, `<|candidate2|>`, and `<|candidate|>` (added_tokens.json).

### Training details:
The PAIRRANKER model was trained using the following configuration:
*   **Algorithm:** The model is trained on a pairwise comparison task. It jointly encodes an input text and a pair of candidate outputs to learn which candidate is better (LLM-BLENDER_paper.pdf, p. 2).
*   **Epochs:** The model was trained for 5 epochs (LLM-BLENDER_paper.pdf, p. 14).
*   **Optimizer:** Adafactor optimizer was used (LLM-BLENDER_paper.pdf, p. 14).
*   **Learning Rate:** A maximum learning rate of 1e-5 with a linear scheduler and a 5% warm-up ratio (LLM-BLENDER_paper.pdf, p. 14).
*   **Batch Size:** 64 (LLM-BLENDER_paper.pdf, p. 14).
*   **Loss Function:** The training uses a loss function based on pairwise comparisons. The paper mentions using Binary Cross-Entropy (BCE) loss and a multi-task classification loss `LQ` based on the sigmoid function (LLM-BLENDER_paper.pdf, p. 4, 5, 14). The configuration file specifies the `loss_type` as `instructgpt` (config.json).
*   **Parameters:**
    *   Dropout: 0.05 (config.json).
    *   FP16 precision was enabled during training (config.json).
*   **Data Sampling:** During training, a sub-sampling strategy was used where 5 pairs of candidates were randomly selected per input to ensure learning efficiency (LLM-BLENDER_paper.pdf, p. 5). The order of candidates within each training pair was shuffled to help the model learn to be consistent (LLM-BLENDER_paper.pdf, p. 6).

## Intended Use
This section outlines the intended applications of the model.

### Primary intended uses:
The PAIRRANKER model is intended to be used as a component within the LLM-BLENDER framework for ensembling multiple Large Language Models (LLMs) (LLM-BLENDER_paper.pdf, p. 1). Its primary purpose is to rank a list of candidate text outputs generated by different LLMs for a given instruction or input prompt. It specializes in distinguishing subtle differences between high-quality candidate texts to identify the best one (LLM-BLENDER_paper.pdf, p. 2).

The model is designed for ranking outputs in instruction-following tasks as well as conventional Natural Language Generation (NLG) tasks such as abstractive summarization, machine translation, and constrained text generation (LLM-BLENDER_paper.pdf, p. 3, 8).

The input-output structure is as follows:
*   **Input:** A single string that concatenates a source text (`x`) and two candidate texts (`yi`, `yj`) using special separator tokens. The format is: `"<s><source> x </s> <candidate1> yi </s> <candidate2> yj </s>"` (LLM-BLENDER_paper.pdf, p. 5).
*   **Output:** The model produces scores that represent the pairwise comparison result, indicating which of the two candidates is predicted to be better (LLM-BLENDER_paper.pdf, p. 4). By comparing all pairs, a full ranking of candidates can be inferred (LLM-BLENDER_paper.pdf, p. 6).

### Primary intended users:
The primary users are researchers and practitioners in the field of Natural Language Processing and AI who are interested in deploying, studying, or improving the performance of LLMs through ensemble methods (LLM-BLENDER_paper.pdf, p. 2).

### Out-of-scope uses:
This model is not a generative model and should not be used to create text. It is a ranking/scoring model designed to evaluate existing text generations. It is intended to be used as part of a larger system (like LLM-BLENDER) and not as a standalone tool for end-users. The model was trained on outputs from specific open-source LLMs and its performance on ranking outputs from closed-source models or models significantly different from its training distribution has not been evaluated.

## How to Use
This section outlines how to use the model. 

The model is a `pairranker` designed to compare two candidate texts for a given source input (config.json).

To use the model for a pairwise comparison, structure the input as a single string:
`"<s><source> source_text </s> <candidate1> candidate_text_1 </s> <candidate2> candidate_text_2 </s>"`
where `source_text`, `candidate_text_1`, and `candidate_text_2` are the respective texts (LLM-BLENDER_paper.pdf, p. 5).

The model will then output logits representing the comparison scores. To obtain a full ranking for N candidates, this process is repeated for all N(N-1)/2 pairs. The resulting pairwise scores can be aggregated to produce a final ranking. The paper proposes three aggregation methods (LLM-BLENDER_paper.pdf, p. 6):
1.  **MaxLogits:** For each candidate, sum the logit differences against all other candidates.
2.  **MaxWins:** For each candidate, count the number of "wins" in pairwise comparisons.
3.  **Bubble Sort:** An efficient method that performs N-1 comparisons to find the best candidate. The `inference_mode` in the model's configuration is set to `bubble` (config.json).

## Factors
This section addresses variables that may impact the model's performance.

### Relevant factors:
The model's performance is influenced by the following factors:
*   **Source LLMs:** The performance depends on the diversity of strengths and weaknesses of the underlying LLMs being ensembled. These variations arise from differences in training data, architectures, and hyperparameters (LLM-BLENDER_paper.pdf, p. 1).
*   **Task Type:** The model is designed for instruction-following and NLG tasks. Its effectiveness may vary across different tasks like summarization, translation, or creative writing.
*   **Response Length:** The model is designed to handle subtle differences, which are often more prominent in shorter responses where other ranking methods may fail (LLM-BLENDER_paper.pdf, p. 4).

### Evaluation factors:
The model was evaluated on its ability to rank candidate outputs from 11 different open-source LLMs, including Vicuna, OpenAssistant, Alpaca, MPT, and others (LLM-BLENDER_paper.pdf, p. 3, 7). The evaluation tasks included general instruction-following (MixInstruct dataset), abstractive summarization (CNN/DailyMail), constrained text generation (CommonGen), and machine translation (WMT18 zh-en) (LLM-BLENDER_paper.pdf, p. 3, 15).

## Metrics
This section describes how the model's performance is evaluated.

### Model performance measures:
The performance of the PAIRRANKER model is assessed by measuring the correlation between its predicted rankings and an oracle ranking provided by GPT-3.5 (termed GPT-Rank). The specific correlation metrics used are (LLM-BLENDER_paper.pdf, p. 8, Table 3):
*   **Pearson Correlation Coefficient**
*   **Spearman's Rank Correlation Coefficient**
*   **Spearman's Footrule Distance**

The overall LLM-BLENDER framework, which utilizes this ranker, is evaluated using standard NLG metrics such as BERTScore, BARTScore, BLEURT, ROUGE, BLEU, and CIDEr, in addition to GPT-Rank (LLM-BLENDER_paper.pdf, p. 6, 7).

### Variation approaches:
To produce a final ranking from the pairwise comparison scores, three aggregation methods are proposed and analyzed (LLM-BLENDER_paper.pdf, p. 6):
*   **MaxLogits:** Aggregates the logit scores from the full N x N comparison matrix.
*   **MaxWins:** Counts the number of wins for each candidate in the comparison matrix.
*   **Bubble Sort:** An efficient iterative approach requiring only N-1 comparisons to find the top candidate.

Experiments show that `MaxLogits` yields the best performance, and it is used as the default aggregator for PAIRRANKER (LLM-BLENDER_paper.pdf, p. 6).

## Evaluation Data
This section provides details about the datasets used to evaluate the model.

### Datasets:
The model was evaluated on the following datasets:
1.  **MixInstruct:** A benchmark dataset created for this research, consisting of a 5k test split. It is a mixture of multiple instruction datasets, including Alpaca-GPT4, Dolly-15K, GPT4All-LAION, and ShareGPT. For each instruction in the test set, it includes candidate responses from 11 open-source LLMs and oracle pairwise comparisons from ChatGPT (GPT-Rank) for evaluation (LLM-BLENDER_paper.pdf, p. 3, Table 1).
2.  **CNN/DailyMail:** A public dataset for abstractive summarization (LLM-BLENDER_paper.pdf, p. 15).
3.  **CommonGen:** A dataset for generative commonsense reasoning (LLM-BLENDER_paper.pdf, p. 15).
4.  **WMT18 (zh-en):** A dataset for Chinese-to-English machine translation (LLM-BLENDER_paper.pdf, p. 15).

### Motivation:
MixInstruct was created to provide a large-scale benchmark for evaluating LLM ensembling methods on instruction-following tasks (LLM-BLENDER_paper.pdf, p. 3). The other datasets were used to assess the model's generalization capabilities on conventional NLG tasks (LLM-BLENDER_paper.pdf, p. 16).

### Preprocessing:
For evaluation, input sequences are created by concatenating the source text and a pair of candidate texts with special separator tokens: `<s><source> x </s> <candidate1> yi </s> <candidate2> yj </s>` (LLM-BLENDER_paper.pdf, p. 5). The tokenizer also performs pre-processing steps such as stripping leading/trailing whitespace and replacing sequences of two or more spaces with a single space (tokenizer.json).

## Training Data
This section provides details about the datasets used to train the model.

### Datasets:
The model was trained on several datasets:
1.  **MixInstruct:** The 100k training and 5k validation splits were used. These contain instructions from various sources and corresponding ground-truth responses (LLM-BLENDER_paper.pdf, p. 3, Table 1). Candidate generations from 11 LLMs were created for these splits.
2.  **Conventional NLG Datasets:** For tasks like summarization (CNN/DailyMail), translation (WMT18), and constrained generation (CommonGen), a special procedure was used. Base models were fine-tuned on one half of the training data, and then used to generate candidates on the other unseen half. This process was repeated for both halves to create a full training set of candidates without the base models having seen the ground truth for the inputs they were generating for (LLM-BLENDER_paper.pdf, p. 15).

### Motivation:
The training data was designed to teach the model to discern quality differences between candidate responses. For MixInstruct and other NLG tasks, supervision was derived from comparing generated candidates against a ground-truth response using metrics like BARTScore (LLM-BLENDER_paper.pdf, p. 4, 7). This approach provides the necessary signal for learning the pairwise ranking task.

### Preprocessing:
The preprocessing steps for training data are the same as for evaluation data. Input sequences are formed by concatenating the source text and a pair of candidate texts with special separator tokens (LLM-BLENDER_paper.pdf, p. 5). Additionally, during training, the order of the two candidates in a pair is shuffled to teach the model to be consistent regardless of position (LLM-BLENDER_paper.pdf, p. 6). A sub-sampling strategy of using 5 pairs per input was employed for training efficiency (LLM-BLENDER_paper.pdf, p. 5).

## Quantitative Analyses
This section presents disaggregated evaluation results.

### Unitary results:
The performance of PAIRRANKER was measured by its correlation with GPT-Rank on the MixInstruct dataset. The results are as follows (LLM-BLENDER_paper.pdf, p. 8, Table 3):
*   **Pearson Correlation:** 46.98
*   **Spearman's Correlation:** 44.98
*   **Spearman's Footrule Distance:** 27.52

These scores were the highest (or best, in the case of Footrule distance) among all tested ranking methods, including baselines like MLM-Scoring, SimCLS, and SummaReranker, as well as individual metrics like BARTScore (LLM-BLENDER_paper.pdf, p. 8, Table 3).

## Caveats and Recommendations
This section lists unresolved issues and provides guidance for users.

### Caveats:
*   **Computational Cost:** The most performant version of the ranker (`MaxLogits` aggregation) requires O(N²) inference calls for N candidates, which can be computationally expensive. While these calls can be parallelized, it is still a significant cost (LLM-BLENDER_paper.pdf, p. 6, 9).
*   **Evaluation Limitations:** The evaluation relies heavily on ChatGPT as a proxy for human judgment (GPT-Rank). The authors acknowledge that automatic metrics and LLM-based evaluation have limitations and that large-scale human evaluation was not feasible for this project (LLM-BLENDER_paper.pdf, p. 10).
*   **Model Consistency:** The model's output can be sensitive to the order of the two candidates in the input pair. The authors found that shuffling the order during training resulted in over 90% consistency during inference, but it is not perfect (LLM-BLENDER_paper.pdf, p. 6, 17).

### Recommendations:
*   **Inference Efficiency:** For users concerned about inference time, the paper recommends using the `Bubble Sort` aggregation method. It requires only N-1 comparisons and achieves strong performance, though it may be slightly less accurate than the full `MaxLogits` comparison (LLM-BLENDER_paper.pdf, p. 17).
*   **Maximizing Performance:** To achieve the best possible ranking performance, users should use the `MaxLogits` aggregation method, which leverages the full pairwise comparison matrix. This can be done efficiently if parallel computation is available (LLM-BLENDER_paper.pdf, p. 17).
*   **Generalization:** The PAIRRANKER model, even when trained on candidates from one set of models, can be effectively transferred to rank outputs from other models, such as GPT-3 (LLM-BLENDER_paper.pdf, p. 16). This suggests it can be used as a general-purpose ranker for a given task.

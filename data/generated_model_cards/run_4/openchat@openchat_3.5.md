## Model Details
This section provides fundamental information about the model, helping stakeholders understand its context and key characteristics.

### Person or organization developing model:
The model and its training framework, OpenChat, were developed by researchers from Tsinghua University, the Institute for AI Industry Research (AIR), and the Shanghai Artificial Intelligence Laboratory (OpenChat Paper, page 1). The authors are Guan Wang, Sijie Cheng, Xianyuan Zhan, Xiangang Li, Sen Song, and Yang Liu (OpenChat Paper, page 1).

### Model date:
The associated research paper was published as a conference paper at ICLR 2024 and the most recent version on arXiv is dated March 16, 2024 (OpenChat Paper, page 1).

### Model version:
The model type is specified as `openchat_v3.2_mistral` (trainer_state.json). This model is an application of the OpenChat framework, which proposes a Conditioned-RLFT (C-RLFT) method to fine-tune language models on mixed-quality data (OpenChat Paper, Abstract). The paper primarily discusses experiments on a 13B parameter model (`openchat-13b`), but the provided repository files correspond to this Mistral-based version (config.json, trainer_state.json).

### Model type:
The model is a `MistralForCausalLM`, a type of Transformer-based language model for causal language modeling (config.json). It is based on the `imone/Mistral_7B_with_EOT_token` model (trainer_state.json).

**Architecture Details (config.json):**
*   **Model Type:** `mistral`
*   **Hidden Size:** 4096
*   **Intermediate Size:** 14336
*   **Number of Hidden Layers:** 32
*   **Number of Attention Heads:** 32
*   **Number of Key-Value Heads:** 8 (Grouped-Query Attention)
*   **Activation Function:** `silu`
*   **Sliding Window Attention:** 4096 tokens
*   **Max Position Embeddings (Context Length):** 8192 tokens
*   **Vocabulary Size:** 32002
*   **Floating Point Precision:** `bfloat16`

**Model Size:**
*   **Total Parameters:** The model has approximately 7 billion parameters, as inferred from its Mistral 7B base (trainer_state.json).
*   **Total Size on Disk:** 14.48 GB (pytorch_model.bin.index.json).

### Training details:
The model was fine-tuned using a novel method called **Conditioned-Reinforcement Learning Fine-Tuning (C-RLFT)** (OpenChat Paper, Abstract). C-RLFT is designed to leverage mixed-quality data without requiring explicit preference labels.

**C-RLFT Framework (OpenChat Paper, Section 3):**
*   The method treats different data sources (e.g., conversations generated by GPT-4 vs. GPT-3.5) as having coarse-grained reward labels (expert vs. sub-optimal).
*   It learns a class-conditioned policy by conditioning the model on the data source via distinct prompt templates (e.g., `GPT4 User: ...` for expert data).
*   The optimal policy is solved through a single-stage, RL-free supervised learning process, which simplifies training compared to traditional RLHF methods. This is formulated as a reward-weighted regression problem.

**Hyperparameters (trainer_state.json, OpenChat Paper, Section 4.1):**
*   **Epochs:** 5
*   **Learning Rate:** 1.25e-05
*   **Learning Rate Schedule:** Cosine schedule, decaying to 10% of the max value.
*   **Optimizer:** AdamW
*   **AdamW Betas:** `beta1`: 0.9, `beta2`: 0.95
*   **AdamW Epsilon:** 1e-05
*   **Weight Decay:** 0.1
*   **Batch Size per GPU:** 10
*   **Training Framework:** DeepSpeed was used for training.

## Intended Use
This section outlines the intended applications of the model.

### Primary intended uses:
The model is intended for instruction-following and conversational chat applications (OpenChat Paper, Section 4.1). It is designed to generate high-quality, helpful responses by leveraging its training on mixed-quality datasets. The model's performance was evaluated on benchmarks assessing its ability to follow instructions (AlpacaEval), engage in multi-turn conversations (MT-bench), and handle various question categories like reasoning, coding, and writing (Vicuna-bench) (OpenChat Paper, Section 4.1).

The model expects input in a specific chat format, where roles are prefixed with `GPT4 Correct User:` and `GPT4 Correct Assistant:`, and turns are separated by `<|end_of_turn|>` (tokenizer_config.json).

### Primary intended users:
The primary intended users are AI researchers and developers interested in building upon or experimenting with open-source language models (OpenChat Paper, Ethics Statement). The project aims to democratize AI research and provide a transparent platform for innovation.

## How to Use
This section outlines how to use the model.

The model should be used in a conversational manner. The input should be formatted according to the model's chat template. For best performance during inference, it is recommended to use the template associated with the "expert" (GPT-4) data source used during training (OpenChat Paper, Section 3.2, 5.3).

**Chat Template (tokenizer_config.json):**
```
{{ bos_token }}{% for message in messages %}{{ 'GPT4 Correct ' + message['role'].title() + ': ' + message['content'] + '<|end_of_turn|>'}}{% endfor %}{% if add_generation_prompt %}{{ 'GPT4 Correct Assistant:' }}{% endif %}
```

**Example Usage with `transformers`:**
```python
from transformers import AutoTokenizer, AutoModelForCausalLM

# The model identifier from the repository
model_name = "openchat/openchat_3.5" 
# Or use the local path if you have downloaded the files
# model_name = "./" 

tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)

# Format the conversation using the chat template
messages = [
    {"role": "user", "content": "Hello, how are you?"},
    {"role": "assistant", "content": "I'm doing great. How can I help you today?"},
    {"role": "user", "content": "Can you explain the importance of open-source AI?"}
]
prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)

# Generate a response
inputs = tokenizer(prompt, return_tensors="pt")
outputs = model.generate(**inputs, max_new_tokens=150)
response = tokenizer.decode(outputs[0], skip_special_tokens=True)

print(response)
```

## Factors
This section addresses variables that may impact the model's performance.

### Relevant factors:
*   **Training Data Quality:** The model's training is explicitly designed around the concept of "mixed-quality data". The performance is influenced by the distinction between "expert" data (from GPT-4) and "sub-optimal" data (from GPT-3.5) (OpenChat Paper, Section 3.1).
*   **Training Data Size:** The relative size of the expert and sub-optimal data subsets can impact performance. The paper shows that expert data, even in limited quantities, is extremely important (OpenChat Paper, Section 5.4).
*   **Prompting Template:** The model is a class-conditioned policy, meaning the prompt template used during inference significantly affects output quality. Using the "expert" (GPT-4) prompt template is crucial for achieving optimal performance (OpenChat Paper, Section 5.3).

### Evaluation factors:
The model's evaluation analyses performance based on the key components of its training methodology:
*   **Coarse-grained Rewards:** The effect of weighting data sources differently (expert vs. sub-optimal) is evaluated in ablation studies (OpenChat Paper, Table 2).
*   **Class-conditioned Policy:** The effect of using different prompt templates for different data sources is also evaluated (OpenChat Paper, Table 2).

## Evaluation Data
This section provides details about the datasets used to evaluate the model.

### Datasets:
The model was evaluated on a suite of standard academic benchmarks (OpenChat Paper, Section 4.1, Appendix C):
*   **AlpacaEval:** A benchmark for instruction-following, consisting of 805 user instructions collected from diverse sources.
*   **MT-bench:** A multi-turn benchmark with 80 high-quality questions across eight topics (writing, roleplay, reasoning, math, coding, etc.) to test conversational and instruction-following abilities.
*   **Vicuna-bench:** A benchmark that evaluates model proficiency across eight question categories, including generic knowledge, commonsense, and coding.
*   **AGIEval:** A collection of human-centric standardized exams (e.g., SAT, LSAT) used to gauge problem-solving and generalization abilities. The evaluation was performed on the English multiple-choice tasks.

### Motivation:
These datasets were chosen because they are widely recognized benchmarks for assessing instruction-following ability and generalization in large language models (OpenChat Paper, Section 4.1).

### Preprocessing:
The evaluation followed the official procedures for each benchmark. For conversational models like OpenChat, the zero-shot prompts for AGIEval were formatted as a user question within the model's specific conversation template (OpenChat Paper, Appendix F).

## Training Data
This section provides details about the datasets used to train the model.

### Datasets:
The model was fine-tuned on the **ShareGPT dataset** (OpenChat Paper, Section 4.1).
*   **Size:** Approximately 70,000 user-shared conversations.
*   **Structure:** The dataset is composed of mixed-quality data, which was separated into two classes for training:
    *   **Expert Data:** ~6,000 conversations generated by GPT-4.
    *   **Sub-optimal Data:** The remaining ~64,000 conversations generated by GPT-3.5.
*   **Source:** The data was collected from `sharegpt.com` (OpenChat Paper, page 2). The specific tokenized data used for this model is referenced as `dataset_openchat3.5/tokenized/openchat_mistral_1017` (trainer_state.json).

### Motivation:
The ShareGPT dataset was chosen because it is a large, publicly available SFT dataset that contains conversations of varying quality. This makes it ideal for the OpenChat framework's goal of effectively fine-tuning models on mixed-quality data without requiring expensive human preference labels (OpenChat Paper, Section 1 & 3).

### Preprocessing:
The training data underwent a specific preprocessing method central to the C-RLFT framework (OpenChat Paper, Section 3.1 & 3.2):
*   **Class-Conditioning:** Each conversation was labeled according to its source (GPT-4 or GPT-3.5).
*   **Template Application:** Distinct conversation templates were applied to each class to create a class-conditioned dataset. For example:
    *   GPT-4 data: `GPT4 User: {question}<|end_of_turn|>GPT4 Assistant:`
    *   GPT-3.5 data: `GPT3 User: {question}<|end_of_turn|>GPT3 Assistant:`
*   **Special Tokens:** A new special token, `<|end_of_turn|>` (ID: 32000), was introduced to differentiate speakers and signal the end of a turn (special_tokens_map.json, OpenChat Paper, Section 3.2).

## Caveats and Recommendations
This section lists unresolved issues and provides guidance for users.

### Caveats:
The developers acknowledge several limitations and areas for future work (OpenChat Paper, Section 7):
*   **Oversimplified Data Quality Assumption:** The method of assigning quality labels (expert vs. sub-optimal) based solely on the data source (GPT-4 vs. GPT-3.5) may be overly simplistic.
*   **Coarse-grained Rewards:** The rewards assigned to data classes are coarse. Future work could explore more fine-grained rewards that reflect the actual quality of each individual data point.
*   **Focus on Instruction-Following:** The model was primarily developed to enhance instruction-following capabilities. Its application for improving reasoning abilities remains an area for future exploration.

### Recommendations:
*   **Use the Correct Inference Prompt:** For optimal performance, users should format their inference requests using the "expert" (GPT-4) prompt template: `GPT4 User: {your prompt}<|end_of_turn|>GPT4 Assistant:`. Using the sub-optimal (GPT-3.5) prompt results in a substantial performance decline (OpenChat Paper, Section 5.3).
*   **Value of Expert Data:** The model's performance is highly sensitive to the presence of high-quality expert data in its training mix, even if the quantity is small. This highlights the importance of data quality over quantity (OpenChat Paper, Section 5.4).

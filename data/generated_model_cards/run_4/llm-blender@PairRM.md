## Model Details
This section provides fundamental information about the model, helping stakeholders understand its context and key characteristics.

### Person or organization developing model:
The model was developed by Dongfu Jiang from Zhejiang University, Xiang Ren from the University of Southern California, and Bill Yuchen Lin from the Allen Institute for Artificial Intelligence (Source: `paper.pdf`, page 1).

### Model date:
The associated academic paper is dated June 30, 2023 (Source: `paper.pdf`, page 1).

### Model version:
This model is the PAIRRANKER module of the LLM-BLENDER framework. LLM-BLENDER is an ensembling framework that also includes a GENFUSER module for fusing outputs. PAIRRANKER is a specialized pairwise comparison model designed to rank outputs from multiple Large Language Models (LLMs) (Source: `paper.pdf`, page 1).

### Model type:
The model is a pairwise ranking model (`pairranker`) designed to compare and rank text outputs from different language models (Source: `config.json`, `paper.pdf`, page 1).

*   **Architecture:** The model's architecture is based on DeBERTa (`microsoft/deberta-v3-large`), a Transformer-based model that uses a cross-attention encoder to jointly encode an input text and a pair of candidate outputs (Source: `config.json`, `paper.pdf`, pages 5, 7).
*   **Size:** The DeBERTa backbone has 400 million parameters (Source: `paper.pdf`, page 7).
*   **Context Length:** The model supports a maximum source text length of 1224 tokens and a maximum candidate text length of 412 tokens (Source: `config.json`).

### Training details:
The PAIRRANKER model was trained using a pairwise comparison method to distinguish subtle differences between candidate outputs from various LLMs (Source: `paper.pdf`, page 1).

*   **Algorithm:** The model is trained on pairs of candidate responses for a given input. It employs a cross-attention mechanism to encode the input and candidates together. The training objective uses a binary cross-entropy (BCE) loss to learn which candidate in a pair is better (Source: `paper.pdf`, pages 4, 14). The configuration also specifies an `instructgpt` loss type (Source: `config.json`).
*   **Parameters and Hyperparameters:**
    *   **Optimizer:** Adafactor optimizer (Source: `paper.pdf`, page 14).
    *   **Learning Rate:** Maximum learning rate of 1e-5 with a 5% warm-up ratio and a linear learning rate scheduler (Source: `paper.pdf`, page 14).
    *   **Epochs:** 5 epochs (Source: `paper.pdf`, page 14).
    *   **Batch Size:** 64 (Source: `paper.pdf`, page 14).
    *   **Dropout:** 0.05 (Source: `config.json`).
    *   **Precision:** The model was trained with FP16 precision (Source: `config.json`).
*   **Training Strategy:** During training, a sub-sampling strategy was used where 5 pairs of candidates were randomly selected per input to ensure learning efficiency (Source: `paper.pdf`, page 5). The order of candidates within each training pair was shuffled to make the model consistent (Source: `paper.pdf`, page 6).

## Intended Use
This section outlines the intended applications of the model.

### Primary intended uses:
The primary intended use of the PAIRRANKER model is to rank a list of candidate text outputs generated by multiple Large Language Models (LLMs) for a given instruction and input. It is a component of the LLM-BLENDER framework, which aims to dynamically ensemble LLMs to produce consistently superior responses (Source: `paper.pdf`, pages 1, 3).

The model takes an input text (`source`) and a pair of candidate texts (`candidate1`, `candidate2`) and determines which candidate is better. By performing pairwise comparisons across all candidates, it can produce a final ranking, allowing for the selection of the best-performing output for a specific task (Source: `paper.pdf`, page 5).

### Primary intended users:
The primary users are researchers and practitioners in the field of Natural Language Processing and AI who are interested in deploying, studying, or improving the performance of LLMs through ensemble learning (Source: `paper.pdf`, page 2).

### Out-of-scope uses:
This model is not a generative model; it does not create text. Its purpose is solely to rank existing text candidates. Using it for tasks other than ranking text outputs, especially in domains not covered by its training data (instruction-following, summarization, translation), may lead to poor performance (Source: `paper.pdf`, pages 1, 15).

## How to Use
This section outlines how to use the model. 

The model expects a single input sequence formed by concatenating the source text, the first candidate, and the second candidate, separated by special tokens. The format is as follows:

`<s><source> [source_text] </s> <candidate1> [candidate_1_text] </s> <candidate2> [candidate_2_text] </s>`

(Source: `paper.pdf`, page 5; `special_tokens_map.json`; `tokenizer_config.json`)

The model will then output logits indicating the relative quality of the two candidates. This process is repeated for all pairs of candidates to generate a comparison matrix, which can then be used to determine the final ranking (Source: `paper.pdf`, page 6).

## Factors
This section addresses variables that may impact the model's performance.

### Relevant factors:
*   **Candidate Quality and Diversity:** The model's performance is dependent on the quality and diversity of the candidate outputs it is ranking. The underlying LLMs used to generate these candidates, including their training data, architectures, and hyperparameters, are key factors (Source: `paper.pdf`, page 1).
*   **Response Length:** The model is designed to be particularly effective at distinguishing subtle differences, which is common in shorter responses where pointwise scoring methods may be insufficient (Source: `paper.pdf`, page 4).
*   **Task Type:** The model has been evaluated on instruction-following tasks as well as conventional NLG tasks like summarization, translation, and constrained generation. Performance may vary across different task types (Source: `paper.pdf`, pages 7, 15).

### Evaluation factors:
The model was evaluated based on:
*   **Ranking Correlation:** Its ability to produce rankings that correlate with an oracle ranker (GPT-Rank) (Source: `paper.pdf`, page 8).
*   **Task-Specific Metrics:** The quality of its top-ranked selections on datasets for summarization (CNN/DM), machine translation (WMT18), and commonsense generation (CommonGen) (Source: `paper.pdf`, page 15).

## Metrics
This section describes how the model's performance is evaluated.

### Model performance measures:
The model's performance is assessed using two main categories of metrics:

1.  **Ranking Correlation Metrics:** To measure how well the model's rankings align with oracle rankings (from ChatGPT), the following metrics are used:
    *   Pearson Correlation Coefficient
    *   Spearman's Correlation
    *   Spearman's Footrule distance
    (Source: `paper.pdf`, page 8, Table 3)

2.  **NLG Evaluation Metrics:** To evaluate the quality of the top-ranked candidate selected by the model, the following metrics are used depending on the task:
    *   **BERTScore, BLEURT, BARTScore:** For general text generation quality (Source: `paper.pdf`, page 7).
    *   **ROUGE-1, ROUGE-2, ROUGE-L:** For summarization tasks (Source: `paper.pdf`, page 15).
    *   **BLEU, CIDEr:** For machine translation and commonsense generation tasks (Source: `paper.pdf`, page 15).
    *   **GPT-Rank:** A metric based on pairwise comparisons from ChatGPT, where rank is determined by the number of wins (Source: `paper.pdf`, page 7).

### Decision thresholds:
The model outputs pairwise comparison scores. These scores are aggregated to produce a final ranking. Three aggregation methods are proposed:
*   **MaxLogits:** Sums the confidence scores for a candidate against all others. This is the default method as it yields the best performance (Source: `paper.pdf`, page 6).
*   **MaxWins:** Counts the number of "wins" for a candidate in pairwise comparisons (Source: `paper.pdf`, page 6).
*   **Bubble Sort:** An efficient method that reduces inference complexity from O(N²) to O(N) (Source: `paper.pdf`, page 6).

### Variation approaches:
The performance of the ranking is analyzed using different aggregation methods (MaxLogits, MaxWins, Bubble Sort) to understand the trade-off between efficiency and accuracy (Source: `paper.pdf`, page 6, 17).

## Evaluation Data
This section provides details about the datasets used to evaluate the model.

### Datasets:
The model was evaluated on several datasets:
1.  **MixInstruct:** A benchmark dataset created for this work, consisting of 5,000 test examples. For each example, it includes outputs from 11 different open-source LLMs and oracle pairwise comparisons from ChatGPT (Source: `paper.pdf`, page 3).
2.  **CNN/DailyMail (CNN/DM):** A standard dataset for abstractive text summarization (Source: `paper.pdf`, page 15).
3.  **CommonGen:** A dataset for constrained commonsense generation (Source: `paper.pdf`, page 15).
4.  **WMT18 (zh-en):** A dataset for Chinese-to-English machine translation (Source: `paper.pdf`, page 15).

### Motivation:
*   **MixInstruct** was created specifically to benchmark and evaluate LLM ensembling methods on instruction-following tasks (Source: `paper.pdf`, page 3).
*   **CNN/DM, CommonGen, and WMT18** were chosen as they are standard public benchmarks for evaluating performance on conventional NLG tasks, allowing for testing the model's generalization capabilities (Source: `paper.pdf`, page 15).

### Preprocessing:
For evaluation, candidate outputs were generated from fine-tuned public checkpoints of various base models (e.g., PEGASUS, T5-large, Opus-MT). For each decoding method (beam search, diverse beam search), 15 candidate generations were produced (Source: `paper.pdf`, pages 15, 16).

## Training Data
This section provides details about the datasets used to train the model.

### Datasets:
The model was trained on the **MixInstruct** dataset, which contains 100,000 training examples and 5,000 validation examples. This dataset is a mixture of instructions from four sources:
*   Alpaca-GPT4 (22,862 examples)
*   Dolly-15K (7,584 examples)
*   GPT4All-LAION (76,552 examples)
*   ShareGPT (3,002 examples)
(Source: `paper.pdf`, page 3, Table 1)

For each example, candidate outputs were generated from 11 popular open-source LLMs (Source: `paper.pdf`, page 3).

### Motivation:
The `MixInstruct` dataset was created to provide a large-scale, diverse set of instruction-following examples with multiple candidate responses, which is necessary for training a robust model to rank and ensemble LLM outputs (Source: `paper.pdf`, page 3).

### Preprocessing:
*   **Candidate Generation:** For each of the 110k examples, 11 candidate outputs were generated using different open-source LLMs (Source: `paper.pdf`, page 3).
*   **Supervision Signal:** For training, the quality of each candidate `yi` was estimated using its similarity to the ground-truth response `y` via metrics like BARTScore (Source: `paper.pdf`, pages 4, 7).
*   **Input Formatting:** The source text `x` and a pair of candidates `yi`, `yj` were concatenated into a single input sequence using special separator tokens: `<s><source> x </s> <candidate1> yi </s> <candidate2> yj </s>` (Source: `paper.pdf`, page 5).
*   **Sub-sampling:** To ensure training efficiency, 5 pairs of candidates were randomly selected for each input during training (Source: `paper.pdf`, page 5).

## Quantitative Analyses
This section presents disaggregated evaluation results.

### Unitary results:
The model's performance was evaluated against several baselines and individual LLMs.

*   **On MixInstruct:** The PAIRRANKER module, using MaxLogits aggregation, achieved a GPT-Rank of 3.20, outperforming the best individual LLM (OpenAssistant, rank 3.90) and other rankers like SimCLS (3.50) and SummaReranker (3.66). Its top-ranked selections were better than or equal to Vicuna in 54.76% of cases and OpenAssistant in 57.79% of cases (Source: `paper.pdf`, page 7, Table 2).
*   **On CommonGen:** When ranking outputs from a T5-large model, PAIRRANKER (max logits) achieved a CIDEr score of 15.86, a 2.45% relative improvement. When ranking outputs from GPT-3, it achieved a CIDEr score of 14.07, a 26.53% relative improvement (Source: `paper.pdf`, page 16, Table 5).
*   **On WMT18 (zh-en):** When ranking outputs from an Opus-MT model, PAIRRANKER (max logits) achieved a BLEU score of 20.47, a 6.12% relative improvement. When ranking outputs from GPT-3, it achieved a BLEU score of 26.36, an 11.65% relative improvement (Source: `paper.pdf`, page 16, Table 6).
*   **On CNN/DailyMail:** When ranking outputs from a PEGASUS-large model, PAIRRANKER (max logits) achieved a ROUGE-1 score of 47.39, a 6.35% relative improvement (Source: `paper.pdf`, page 15, Table 4).

## Caveats and Recommendations
This section lists unresolved issues and provides guidance for users.

### Caveats:
*   **Efficiency:** The best performance from PAIRRANKER is achieved with the `MaxLogits` aggregation method, which requires O(N²) pairwise comparisons for N candidates. While these can be executed in parallel, it is computationally intensive. A more efficient `Bubble Sort` method is also proposed, which requires only O(N) comparisons but may result in slightly lower performance (Source: `paper.pdf`, pages 9, 17).
*   **Evaluation Limitations:** The evaluation relies on automatic metrics and ChatGPT-based comparisons (GPT-Rank) rather than large-scale human evaluation, which was not feasible due to the number of models and candidates (Source: `paper.pdf`, page 10).
*   **Model Consistency:** Due to the nature of the input format, the order of candidates in a pair can affect the output. The developers found that shuffling the candidate order during training resulted in over 90% consistency during inference (Source: `paper.pdf`, page 17).

### Recommendations:
*   For users prioritizing performance, the `MaxLogits` aggregation method is recommended, and the O(N²) inferences can be parallelized (Source: `paper.pdf`, pages 6, 17).
*   For users prioritizing efficiency, the `Bubble Sort` aggregation method is recommended, as it achieves high performance with only N-1 comparisons (Source: `paper.pdf`, page 17).
*   The model is more effective at classifying candidates with a larger absolute rank difference, as they are more distinct (Source: `paper.pdf`, page 17).
